# Domain Model

## Overview
This document describes the meaning, key data points, and business rules of each core entity in the system.

## AdminUser
- Purpose: Represents the authenticated Coach/Admin who operates the system.
- Key Attributes: identifier, full name, email or username, password hash, role, created date, last login date.
- Validation Rules: Credentials must be unique and authentication data must be stored securely.
- Relationships: Associated with actions such as match week creation, recommendation generation, and audit records.

## Player
- Purpose: Represents a squad member in the single club environment.
- Key Attributes: identifier, full name, preferred position, secondary position, squad number, dominant foot, age, status, active flag.
- Validation Rules: Each player should have at least one valid football position and must belong to the single club context implicitly.
- Relationships: Linked to many weekly performance records and many recommendation player records.

## MatchWeek
- Purpose: Represents one weekly planning cycle for which player conditions are recorded and one recommendation is produced.
- Key Attributes: identifier, week label, opponent label if used later, match date, status, created by, created date.
- Validation Rules: A match week must be uniquely identifiable for weekly planning and must exist before weekly performance records can be saved.
- Relationships: Linked to many weekly performance records and one normal final recommendation in version 1.

## WeeklyPerformance
- Purpose: Stores weekly condition and readiness information for a player in a specific match week.
- Key Attributes: identifier, player reference, match week reference, training rating, fitness, fatigue, morale, injury status, suspension status, availability, coach notes.
- Validation Rules: Values must remain within defined ranges, status fields must be consistent, and one player should not have duplicate weekly records for the same match week.
- Relationships: Belongs to one player and one match week.

## Formation
- Purpose: Represents an approved tactical structure that the system may recommend.
- Key Attributes: identifier, formation code, defensive line count, midfield count, attacking count, active flag.
- Validation Rules: Only approved formations are allowed in version 1.
- Relationships: Used by recommendation records.

## Recommendation
- Purpose: Stores the weekly lineup and formation recommendation outcome.
- Key Attributes: identifier, match week reference, formation reference, generation timestamp, generated by, summary explanation, rule-based score context, optional ML support summary, status.
- Validation Rules: A recommendation must belong to one match week and include an explanation suitable for review.
- Relationships: Belongs to one match week, uses one formation, and owns many recommendation player records.

## RecommendationPlayer
- Purpose: Captures the player-level outcome of a recommendation.
- Key Attributes: identifier, recommendation reference, player reference, selected flag, starting role, computed score, selection reason, exclusion reason, rank order.
- Validation Rules: A selected player should have a starting role; an excluded player should have a meaningful exclusion reason when relevant.
- Relationships: Belongs to one recommendation and references one player.

## AuditLog
- Purpose: Preserves traceability for sensitive or important system actions.
- Key Attributes: identifier, admin user reference, action type, entity type, entity reference, timestamp, description.
- Validation Rules: Audit entries should capture enough context to support review without storing unnecessary sensitive data.
- Relationships: Linked to one AdminUser when an authenticated action occurs.

## Domain Rules to Lock In
- The system supports one club globally.
- The only interactive user role in version 1 is Coach/Admin.
- Recommendations are generated per match week.
- Rule-based scoring is the primary recommendation engine.
- Optional ML output is supportive and does not replace rule-based reasoning.
- Injured, suspended, or unavailable players cannot be recommended as starters.
- If fewer than 11 eligible players exist, the system must not produce a normal starting lineup recommendation.
